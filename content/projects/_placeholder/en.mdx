## The approach

The technical approach section covers architecture decisions and the reasoning behind key choices. Why this stack over alternatives? What constraints shaped the design? What tradeoffs were accepted and why?

For example: choosing a static export over a server-rendered deployment reduces infrastructure complexity while still enabling fast page loads via CDN. The tradeoff is that data-heavy or personalized features require client-side fetching, which is acceptable for a portfolio site but not for a transactional application.

### Architecture overview

A brief description of the system's key components and how they connect:

- **Frontend**: Next.js App Router with static export, deployed to Vercel
- **Content**: File-based MDX content model with `gray-matter` for frontmatter parsing
- **i18n**: Route-based locale switching via `next-intl` — every route exists at `/en/` and `/id/`
- **Styling**: Tailwind CSS v4 with design tokens defined in CSS `@theme`
